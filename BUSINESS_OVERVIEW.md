# AI-Chatbotee 业务说明（模块价值与意义）

本文档从“业务价值/用户收益”的视角介绍 AI-Chatbotee 各模块的意义、典型使用场景，以及模块之间如何组合形成完整的工作方式。

## 1. 产品定位（What & Why）

AI-Chatbotee 是一个“可配置、可组合、可沉淀”的 AI 工作台：

- **可配置**：统一管理多家 LLM（OpenAI/Anthropic/Ollama/Gemini/自定义等）与 MCP 工具源（含 OAuth）。
- **可组合**：把对话、工具调用、工作流、终端、本地文件与资料库检索组合成可复用的任务流。
- **可沉淀**：把“对话过程、工具调用、工作流、研究材料、圆桌讨论、技能包”等沉淀为可复盘、可复用的资产。

面向的核心用户：

- 需要**把 AI 变成可落地产出**的个人/团队（产品、运营、研究、开发、内容生产、咨询等）。
- 需要**多模型、多工具、多任务并行**，并希望把方法论与成果结构化沉淀的人。

## 2. 核心业务目标（解决什么问题）

- **降低使用门槛**：把分散的模型/工具/资料整合进一个桌面工作流。
- **提升产出质量与一致性**：通过人设/系统提示词、可复用工作流、技能包、研究资料库，让输出更稳定。
- **提升效率**：工具调用自动化、研究检索、终端辅助、并行多智能体讨论，减少重复劳动。
- **增强可追溯性**：保留会话、消息、总结、工具执行记录，让决策与产出可复盘。

## 3. 典型用户旅程（怎么用）

### 3.1 “资料 → 任务 → 产出”的闭环（Research 模式）

1. 上传资料（URL/文件/目录/图片）作为来源（sources）。
2. 进行检索与引用，形成结构化的研究上下文。
3. 将任务拆分（Split）或分配给课题组成员（Assign，选定 agents）。
4. 产出：研究进展 Markdown、引用证据、结论与行动清单；必要时导出。

### 3.2 “爬取数据 → 人设/上下文 → 对话生产”

1. 通过 Crawler 抓取网页/模块化数据并标准化。
2. 一键将某条内容设为“系统提示词/人设”，或插入为对话内容。
3. 在聊天中使用指定 LLM + 可选 MCP 工具完成写作/分析/执行建议。

### 3.3 “多智能体协作 → 汇总决策”（圆桌会议）

1. 从 Agents 列表选择多个智能体加入圆桌会议。
2. 会议中进行讨论与多回答收敛（选择最佳响应）。
3. 输出：会议纪要、方案对比、共识结论；可反哺为技能包/工作流。

## 4. 模块与功能：业务价值清单

下面按“模块 → 业务价值 → 典型场景 → 产出/沉淀”描述。

### 4.1 智能对话（Chat / Sessions）

- **业务价值**
  - 提供最自然的人机协作入口：快速提问、迭代、确认与交付。
  - 会话与消息持久化，支持长期任务推进与复盘。
- **典型场景**
  - 需求澄清、内容生成、代码/方案评审、调研总结、计划制定。
- **产出/沉淀**
  - 会话（sessions）、消息（messages）、总结（summaries）、多模态内容（图片/视频/音频）。

### 4.2 LLM 配置管理（LLM Config）

- **业务价值**
  - 把“模型选择/密钥/地址/模型参数”从业务逻辑中抽离，降低切换成本。
  - 团队可用统一的配置模板，减少环境差异导致的不稳定输出。
- **典型场景**
  - 不同任务用不同模型：创作/推理/长上下文/成本优化。
- **产出/沉淀**
  - LLM 配置资产（可导入导出，便于迁移与协作）。

### 4.3 MCP 工具调用（MCP Config + Tool Calling）

- **业务价值**
  - 让 AI 不止“会说”，还能“会做”：调用外部工具/系统完成操作或查询。
  - OAuth 支持使受保护资源（如 Notion 等）可安全接入。
- **典型场景**
  - 内容/知识库/业务系统的查询与写入、自动化操作、资料采集。
- **产出/沉淀**
  - MCP 服务器配置、工具列表、工具调用结果（可在会话中记录/复盘）。

### 4.4 Workflow 编辑与运行（Workflow / WorkflowEditor）

- **业务价值**
  - 将重复任务固化为“可复用流程”：减少每次都从 prompt 重新开始的成本。
  - 支持把多个节点（输入/LLM/工具/终端/可视化）串成明确步骤，提升稳定性与可解释性。
- **典型场景**
  - 周报/月报生成、竞品分析模板化、数据→结构化→输出的流水线。
- **产出/沉淀**
  - 工作流定义（nodes/connections）、执行结果与日志（在执行记录中体现）。

### 4.5 感知组件（@ 选择 MCP/Workflow/SkillPack）

- **业务价值**
  - 把“选择一个能力组件”融入输入框交互，降低工具/工作流的调用摩擦。
  - 强制“单次只选择一个感知组件”有助于控制复杂度与产出一致性。
- **典型场景**
  - 某次对话固定用某个 workflow 或某个 MCP 工具集来完成任务。
- **产出/沉淀**
  - 消息与执行记录关联（可追踪某次产出来自哪个组件）。

### 4.6 技能包（SkillPack）

- **业务价值**
  - 将一段高质量对话/方法论提炼成“可复用能力片段”，减少团队知识流失。
  - 技能包可再分配给会话/智能体，形成组织级的 prompt/策略资产。
- **典型场景**
  - 把一次成功的项目复盘、写作风格、评审清单固化为技能包。
- **产出/沉淀**
  - 技能包条目（名称、总结、来源消息集合）、分配关系（assignment）。

### 4.7 Crawler（网页采集与标准化）

- **业务价值**
  - 将外部网页/内容源转换为可用的结构化输入，减少手工复制与清洗。
  - 与“人设/上下文插入”联动，快速把采集内容变成可用上下文与约束。
- **典型场景**
  - 竞品页面抓取、文档/公告采集、结构化条目抽取用于后续分析/生成。
- **产出/沉淀**
  - 模块（module）与批次（batch）数据、标准化后的 items，可复用。

### 4.8 圆桌会议（Round Table / 多智能体协作）

- **业务价值**
  - 用多智能体在同一议题上并行思考，获得多角度方案，提升决策质量。
  - 支持“选择最佳回答/收敛”，降低多答案带来的噪音。
- **典型场景**
  - 方案评审、角色扮演（产品/技术/运营/法务）、风险评估、共识形成。
- **产出/沉淀**
  - 会议（round table）、参与者、消息与候选回应、最终选择结果。

### 4.9 Research（资料库 + 检索 + 任务拆分/分工）

- **业务价值**
  - 把“资料采集、索引、检索、引用、写作”串为一体，提升研究效率与可信度。
  - 支持把研究任务拆分为并发窗口或分配给课题组成员（agents），提高吞吐。
- **典型场景**
  - 行业研究、技术选型、论文/报告整理、客户/项目背景梳理。
- **产出/沉淀**
  - sources（URL/文件/目录）、检索结果（带 snippet/score）、研究进展 Markdown、研究员/成员配置（前端本地映射）。

### 4.10 内嵌 Terminal（本地执行 + LLM 补全）

- **业务价值**
  - 把“建议”落到“执行”：一边对话一边跑命令，减少上下文切换。
  - LLM 补全与历史记录提升命令行效率，尤其在项目操作、脚本调试、数据处理时明显。
- **典型场景**
  - 开发调试、日志分析、批处理脚本、仓库操作、环境排障。
- **产出/沉淀**
  - 终端会话、命令历史、可选的 LLM 补全配置。

### 4.11 智能体（Agents）与导入导出

- **业务价值**
  - 智能体 = 具备头像/人设/默认 LLM/技能包等配置的“可复用角色”，可用于圆桌与 research 分工。
  - 导入导出使团队共享角色配置、跨环境迁移更轻松。
- **典型场景**
  - 固定角色：市场研究员、技术 reviewer、写作编辑、数据分析师等。
- **产出/沉淀**
  - agent 会话、agent 配置包（可导入导出）。

## 5. 数据资产与可复用性（项目长期价值）

该项目的长期价值在于：把一次性对话转化为组织可复用的资产，主要包括：

- 配置资产：LLM configs、MCP servers、OAuth tokens
- 过程资产：sessions/messages/summaries、message executions、round table 讨论记录
- 方法资产：workflows、skill packs
- 资料资产：crawler batches、research sources/documents

## 6. 成功指标（如何衡量业务价值）

建议从以下维度衡量（可作为后续埋点/报表方向）：

- **效率**：完成同类任务的平均时间、复用工作流/技能包的次数、工具调用节省的手工步骤数
- **质量**：输出一致性（模板化程度）、决策收敛速度（圆桌选择最佳答案时间）
- **沉淀**：技能包数量与分配覆盖率、工作流数量与复用率、研究 sources 与引用次数
- **稳定性**：失败率（MCP/LLM 调用失败、超时）、可重试成功率

## 7. 风险与注意事项

- **敏感配置**：严禁提交真实 API Key、数据库口令、OAuth secret；建议使用环境变量或本地私有覆盖。
- **数据合规**：上传的研究资料、爬取内容与工具接入数据需遵守所在组织与目标站点的合规要求。
